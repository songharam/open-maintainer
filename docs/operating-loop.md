# Maintainer Operating Loop

Open Maintainer Workbench is designed around a repeatable weekly loop, not a one-time generated report.

## Goal

Each cycle should turn repository data into a small amount of maintainer action:

- one clearer issue queue
- one reviewed pull request or review checklist
- one contributor-facing task
- one reusable weekly report or release note draft

## Weekly Loop

1. Open the workbench with `?demo=sample` or a public GitHub repository URL.
2. Review the issue triage counts and pick the highest-risk bug or docs gap.
3. Use the priority brief to choose the next maintainer action.
4. Review one pull request with the PR checklist.
5. Promote one `good first issue` candidate or make one task contributor-ready.
6. Copy one generated artifact into the real workflow: issue comment, PR review, release note, README, CONTRIBUTING guide, or weekly update.
7. Check `Readiness`, `Reviewer`, `Launch`, and `Export` before sharing progress externally.

## Done Criteria

A loop is complete when:

- CI is green.
- One generated artifact was used in the real repository workflow.
- One issue, PR, release note, or contributor task is clearer than before.
- The Markdown export reflects the latest repository state.

## Why This Matters

The support case is stronger when the project shows a repeatable maintainer behavior. The loop demonstrates that API credits would improve an ongoing workflow: better triage, review risk summaries, release notes, contributor onboarding, and weekly reporting.
